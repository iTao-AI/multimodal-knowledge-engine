## What is new

The repository now includes a deterministic local knowledge proof that:

- starts the real stdio MCP server and uses the existing MCP tools;
- ingests two repository-authored synthetic text-layer PDFs through observable Runs;
- requires both Runs to reach the published state;
- binds returned page Evidence to the fixed fixture content and valid locators;
- searches active Publications and requires an Evidence hit;
- asks an evidence-supported question and requires a cited answer;
- asks an unsupported question and requires zero-citation `insufficient_evidence`;
- renders only aggregate results, without local paths, transient IDs, source text, or tracebacks.

Run it offline without model downloads:

```bash
UV_OFFLINE=1 uv run python scripts/local_knowledge_proof.py
```

The detailed workflow is in
[Run The Local Knowledge Proof](../how-to/run-local-knowledge-proof.md).

## Runtime boundary

This patch release does not change the MCP schemas, CLI/MCP application contract, Search, Ask,
Publication lifecycle, owner startup, or retrieval runtime default. The shipped runtime remains
local CLI plus stdio MCP with SQLite domain truth and `cjk-active-scan-overlap-v1` as the default
owner-startup CJK strategy.

E3-C dense, E3-D RRF, and E3-E relevance-gate/reranker results remain comparison-only evidence.
They are not runtime features in `v0.1.1`.

## Known limitations

- The local knowledge proof is a deterministic two-document synthetic case, not a general
  retrieval-quality benchmark.
- OCR, layout-aware table extraction, long-video processing, natural-language answer generation,
  HTTP, UI, and public API adapters are not included.
- Optional transcription and embedding paths remain explicit operator actions and are not required
  by this release proof.

## Upgrade path

`v0.1.1` is contract-compatible with `v0.1.0`. Use `0.2.0` for contract-expanding work such as
dense or hybrid retrieval promotion, reranker execution, query rewrite, segmentation, OCR, HTTP,
UI, or public API adapters.
