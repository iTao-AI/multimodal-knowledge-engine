# v0.1.2 Release Notes

`v0.1.2` makes the additive Evidence provenance contract and external source-pack consumer proof
visible as one runtime-neutral small release. Strict `mke.evidence_ref.v1` values connect Evidence
to source bytes, active Publication revision, producing Run, and page or timestamp locator without
changing the existing Search, Ask, Publication, Run, or MCP behavior.

## Evidence And Consumer Proof

- `list_libraries_v1`, `search_library_v1`, and `ask_library_v1` expose additive versioned
  read-only MCP responses.
- Provenance validation fails closed when the portable source, Publication, Run, or locator graph
  is inconsistent.
- The external source-pack proof uses the official MCP SDK, does not import MKE or inspect SQLite,
  and proves the same wheel in fresh Python 3.12 and Python 3.13 environments.
- Exact source-pack fingerprint mapping is reproduced in fresh stores.
- Bounded controller output, deadlines, stable redacted failures, cancellation, subprocess
  cleanup, owner lifecycle, and atomic transition boundaries remain hardened.
- Local candidate handoff emits one exact wheel and `mke.candidate_artifact_receipt.v1` bound to
  the clean source commit and proven wheel SHA-256.

## Downstream Validation Boundary

An independent consumer validated a pre-release candidate from synthetic fixtures and a strict
receipt in [Night Voyager PR #21](https://github.com/iTao-AI/night-voyager/pull/21). The candidate
was built from source commit `16fae017ced5fe67da3fae4a01f26e9e9f1084aa` and exercised the real
integration boundary.

This evidence did not validate the final v0.1.2 wheel and does not prove production adoption,
hosted deployment, or real-user outcomes. Night Voyager remains an independent consumer and is
not an MKE CI dependency. An identity-only release does not require a downstream lock update. If
runtime behavior or a consumed contract changes, this equivalence no longer applies.

## Runtime And Evaluation Boundary

Existing `v0.1.1` capabilities remain: observable Runs, active-Publication-only Search and Ask,
page or timestamp Evidence, deterministic product proof, local knowledge proof, CLI, and stdio MCP.
The runtime default remains `cjk-active-scan-overlap-v1`.

E3-C dense, E3-D RRF, and E3-E relevance-gate/reranker remain comparison-only evidence. PDF OCR
Phase 0, layout-aware extraction, retrieval runtime promotion, query rewrite, HyDE, HTTP, UI,
service adapters, hosted deployment, and PyPI publication are excluded.

## Verification

Run the repository, proof, release-presentation, and exact installed-wheel gates:

```bash
uv run pytest -q
uv run ruff check .
uv run pyright
uv build
uv run mke proof run
uv run mke demo --verify
UV_OFFLINE=1 uv run python scripts/local_knowledge_proof.py
UV_OFFLINE=1 uv run python scripts/evidence_provenance_proof.py
uv run python scripts/release_presentation_audit.py --root .
uv run python scripts/release_consumer_smoke.py \
  --wheel dist/multimodal_knowledge_engine-0.1.2-py3-none-any.whl --json
```

Run the same-wheel source-pack gate with explicit interpreters and local candidate output:

```bash
candidate_parent="$(mktemp -d)"
candidate_output="${candidate_parent}/mke-v0.1.2-candidate"
UV_OFFLINE=1 uv run python scripts/consumer_source_pack_proof.py \
  --python "$(command -v python3.12)" \
  --python "$(command -v python3.13)" \
  --candidate-output "${candidate_output}" \
  --json
```

The candidate output is local maintainer evidence, not a GitHub Release asset, PyPI artifact,
deployment, or production-adoption proof. Tagging and GitHub Release publication require separate
authorization after the complete gate passes on the exact final `main` commit.
