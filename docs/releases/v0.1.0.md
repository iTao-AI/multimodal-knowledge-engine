# v0.1.0 Release Notes

`v0.1.0` is the first public small version of Multimodal Knowledge Engine.

Release identity:

- Package version: `0.1.0`
- GitHub Release metadata records the final tag and target commit after Stage 2 consumer smoke
  passes.
- This document describes release scope and verification before publication; it does not predeclare
  a future tag target.

## What ships

MKE ships as a local-first, Agent-callable Evidence engine. The verified slice can:

- ingest text-layer PDFs and the documented short local video fixture through observable Runs;
- publish only successful output and keep failed or partial processing out of Search;
- search active Publications and return page or timestamp Evidence;
- answer with cited Evidence or `insufficient_evidence`, without calling an LLM;
- expose CLI and stdio MCP tools over one canonical application contract;
- run deterministic product proof commands with `mke proof run` and `mke demo --verify`;
- use `cjk-active-scan-overlap-v1` as the default owner-startup CJK retrieval strategy.

The runtime remains local CLI plus stdio MCP. HTTP and workspace UI are planned but not shipped.

not change Search, Ask, MCP, owner startup, Publication, ingestion, or the runtime default.

## Optional extras

The core release gate does not require model downloads.

- `[transcription]` enables optional cache-only faster-whisper runtime paths after explicit model preparation.
- `[embedding]` enables optional comparison-only local embedding evaluation after explicit model preparation.

Model acquisition remains opt-in and outside normal proof, demo, CLI ingest, and MCP execution.

## Known limitations

- The CJK runtime is bounded lexical active-scan evidence over a small public Chinese page-level corpus.
- Dense, RRF, and relevance-gate/reranker results are useful comparison evidence, not shipped behavior.
- OCR, layout-aware table extraction, long-video processing, natural-language answer generation, HTTP, and UI are not included.
- Stage 2 still needs installed-package consumer smoke before tag and GitHub Release publication.

## Upgrade path

Use `0.1.x` for release polish, documentation fixes, small bug fixes, and non-contract-breaking
developer experience improvements.

Use `0.2.0` for contract-expanding work such as dense or hybrid retrieval promotion, reranker
execution, query rewrite, segmentation, OCR, HTTP, UI, or public API adapters.
