# v0.1.3 Release Closeout Design

Status: approved for implementation and publication, subject to every gate in this design.

Date: 2026-07-17

## Context

The current `main` line already contains two reviewed and merged increments after `v0.1.2`:

1. PDF OCR Phase 0 viability evidence over a fixed public-safe synthetic corpus; and
2. deterministic Compiled Library Export with an installed-wheel external consumer proof.

The package identity remains `0.1.2`, so neither increment is represented by a published version.
This release advances the small-version line by `0.0.1` to `v0.1.3`. It does not introduce a new
runtime architecture, hosted service, or dependency family.

## Release Decision

`v0.1.3` is a small feature release led by Compiled Library Export.

The release headline is:

> MKE can deterministically export active Publications as portable Markdown with exact page or
> timestamp Evidence provenance, validated through an installed-wheel external consumer proof.

PDF OCR Phase 0 is secondary evaluation evidence. It establishes bounded local viability and a
planning baseline for a separate production implementation. It is not a public OCR ingest feature.

The release may describe portable Markdown as suitable input for downstream Agent or knowledge
compiler workflows. It must not claim verified compatibility with LLM Wiki or any other named
downstream compiler until that isolated compatibility proof exists.

## Goals

- Publish package and module version `0.1.3` from the exact verified `main` commit.
- Make Compiled Library Export the primary release capability.
- Preserve strict `mke.evidence_ref.v1` authority in exported JSONL while treating Markdown as a
  readable derivative.
- Record PDF OCR Phase 0 accurately as evaluation-only evidence.
- Prove the exact candidate wheel through installed smoke, the existing source-pack consumer, and
  the compiled-export consumer on Python 3.12 and 3.13.
- Preserve the existing runtime, retrieval, MCP, dependency, and database boundaries.
- Publish an annotated `v0.1.3` tag and non-draft, non-prerelease GitHub Release after the exact
  merged-main gate passes.
- Verify the public source archive and record durable post-release provenance.

## Included Capabilities

### Compiled Library Export

The release includes the read-only CLI surface:

```text
mke --db <library.sqlite3> library export --output <new-child-directory> [--json]
```

The exporter:

- snapshots active Publications from SQLite without changing domain state;
- writes canonical portable Markdown under `sources/`;
- writes matching `mke.evidence_ref.v1` JSONL under `evidence/`;
- publishes `export-manifest.json` last as the commit marker;
- validates exact inventory, content digests, path identity, and bounded cleanup before success;
- never includes original Source bytes or claims layout reconstruction; and
- fails closed on invalid provenance, oversized content, ambiguous ownership, or write races.

The generic proof builds the current candidate wheel, installs the same wheel in fresh Python 3.12
and 3.13 environments, invokes the installed producer, and validates the result with a standalone
consumer that imports no MKE code and reads no SQLite database.

### PDF OCR Phase 0 Evidence

The repository includes a closed Phase 0 protocol, deterministic synthetic fixtures, isolated
candidate adapters, package/model/startup receipts, a scorecard, and an installed consumer proof.
The accepted planning baseline is:

- provider: `ppocrv6-medium-cpu-spike-v1`;
- profile: `phase0-200dpi-plain-text-v1`.

PaddleOCR-VL 1.6 remains a validated Phase 0 comparison candidate. Apple Vision remains a measured
local baseline. Measurements are facts for the fixed Phase 0 protocol, not general OCR quality or
production resource claims.

The committed OCR receipts and scorecard remain historical evidence bound to their recorded wheel,
package, model, protocol, and runtime identities. A version-only release must not rewrite those
facts to pretend they were generated by `0.1.3`.

### Existing Capabilities Carried Forward

The release retains the `v0.1.2` Evidence lifecycle, active Publication Search, evidence-only Ask,
strict portable Evidence provenance, CLI and stdio MCP application contract, local knowledge proof,
source-pack proof, owner lifecycle hardening, and `cjk-active-scan-overlap-v1` runtime default.

|---|---|---|
| Compiled Library Export | Released read-only CLI capability with deterministic Markdown and exact EvidenceRef sidecars. | Markdown is derivative; JSONL is machine authority; no Source bytes or layout reconstruction. |
| Installed export proof | The same candidate wheel passes Python 3.12 and 3.13 external consumer verification. | Local synthetic proof, not adoption, hosted integration, or business impact. |

## Non-Goals

This release does not:

- add production OCR to normal PDF ingest;
- expose an OCR provider, model, flag, configuration, CLI, MCP, or API surface;
- approve OCR quality or resource service-level objectives;
- add layout reconstruction, table/formula authority, or rich document IR;
- verify LLM Wiki compatibility;
- add HTTP, UI, multi-tenant service, RBAC, or hosted deployment;
- change the stdio MCP schema or shared PublicError allowlist;
- promote dense, RRF, reranker, query rewrite, or HyDE into the runtime;
- publish to PyPI or another package registry;
- upload the local candidate wheel or receipt as a GitHub Release asset; or
- deploy any service.

## Version And Historical Evidence Rules

The current release identity becomes `0.1.3` in:

- `pyproject.toml`;
- `src/mke/__init__.py`;
- package/bootstrap version tests;
- current release smoke and presentation audit constants;
- current README, documentation index, release notes, changelog, and release verification commands.

Historical release records remain historical:

- `docs/releases/v0.1.2.md` remains byte-for-byte a `v0.1.2` record;
- the completed `v0.1.2` design, plan, reviews, tag identity, archive digest, and publication facts
  are not rewritten as current facts;
- version literals inside historical OCR receipts, frozen fixtures, compatibility tests, or
  historical proof examples are changed only when a test explicitly proves they represent current
  package identity rather than recorded historical authority.

`uv.lock` may change only through the mechanical project-version update. Dependency names,
versions, sources, markers, hashes, and extras must remain semantically unchanged.

## Four-Stage Release Architecture

### Stage 1: Release Candidate PR

Create an isolated branch from the exact current `origin/main`. Define the `0.1.3` identity in
tests first, then update package identity, release presentation, release notes, and exact release
verification instructions. Any frozen retrieval evaluation changes must be limited to a
validator-proven identity/provenance closure.

No tag or GitHub Release exists at this stage.

### Stage 2: Clean Candidate Verification

Run all repository gates from a clean committed candidate. Build fresh artifacts and generate one
local candidate-output directory with:

- exactly one `multimodal_knowledge_engine-0.1.3-py3-none-any.whl`; and
- exactly one `candidate-artifact-receipt.json` using
  `mke.candidate_artifact_receipt.v1`.

The receipt must bind the exact clean source commit, wheel filename, wheel bytes, wheel SHA-256,
package version, and source-pack proof digest.

Run the installed release smoke against that exact receipt-bound wheel. Independently run the
compiled Library export proof with Python 3.12 and 3.13 from the same clean source commit. Its
`proof_input_wheel_sha256` must equal the candidate receipt's exact wheel SHA-256. Equality of only
the filename or version is insufficient.

Any tracked write after candidate generation invalidates the candidate evidence and requires a
fresh Stage 2 run.

### Stage 3: Exact Merged-Main Gate

After the release-candidate PR merges, update local `main` and prove:

- local `main == origin/main`;
- the worktree is clean;
- the merge commit contains the reviewed feature tree; and
- every Stage 1 and Stage 2 gate passes from that exact merged commit.

Rebuild the wheel, receipt, source-pack result, compiled-export result, and installed smoke. Do not
reuse branch artifacts or logs.

### Stage 4: Annotated Tag, GitHub Release, And Public Archive Smoke

Only after Stage 3 passes:

1. create annotated tag `v0.1.3` targeting the exact verified merge commit;
2. push the tag;
3. create a non-draft, non-prerelease GitHub Release using the reviewed release notes;
4. upload no additional assets;
5. download the public GitHub-generated source archive into a clean temporary directory;
6. verify archive digest, version identity, locked installation, product proof, demo, local
   knowledge proof, Evidence provenance proof, and a real `library export` plus standalone
   consumer smoke; and
7. record tag object SHA, peeled target, publication time, archive name/bytes/SHA-256, and smoke
   results in a docs-only post-release closeout.

The public archive smoke validates the released source archive. It does not need to repeat model
downloads or the Phase 0 real-provider run.

## Evaluation Identity Closure

Version, release documentation, audit, test, or workflow byte changes may invalidate frozen
retrieval provenance. If canonical validators report drift, use the existing repository rebinding
and transaction path to refresh only the dependency-closed identity fields.

The closure must prove normalized semantic equality before and after for every affected layer.
It must not change corpus, fixtures, qrels, queries, observations, ordered results, metrics,
thresholds, gates, diagnostics, selected profile/candidate, status, or verdict.

OCR receipts and `benchmarks/ocr/phase0-scorecard.json` are not part of this identity refresh and
must remain byte-identical.

If the required closure exceeds the repository's established allowlist or changes evaluation
semantics, stop for authority review.

## Documentation Contract

The release-candidate PR updates:

- `README.md` and `README_CN.md` so `v0.1.3` is current and Compiled Library Export is the lead;
- `docs/README.md` so `v0.1.3` release notes and current proof boundaries are discoverable;
- `CHANGELOG.md` with an `0.1.3` entry;
- new `docs/releases/v0.1.3.md`;
- `docs/how-to/verify-release.md` with current candidate/final-main commands while preserving
  completed historical release records;
- compiled-export proof wording so running a proof is separated from publishing a release; and
- current consumer-source-pack candidate wording from `v0.1.2` to `v0.1.3` where it describes the
  current checkout rather than historical evidence.

The release presentation audit must fail closed on:

- version disagreement;
- stale statements that Compiled Library Export is only a future release candidate;
- claims of verified LLM Wiki compatibility;
- claims of production OCR, OCR runtime promotion, general OCR quality, or reconstructed layout;
- claims of hosted deployment, production adoption, business impact, PyPI publication, or extra
  GitHub Release assets; and
- accidental mutation of the current retrieval runtime boundary.

## Required Verification

Before authoritative review and again on final committed candidate:

```bash
UV_OFFLINE=1 uv run pytest -q
UV_OFFLINE=1 uv run ruff check .
UV_OFFLINE=1 uv run pyright
UV_OFFLINE=1 uv build
UV_OFFLINE=1 uv run mke proof run
UV_OFFLINE=1 uv run mke demo --verify
UV_OFFLINE=1 uv run python scripts/local_knowledge_proof.py
UV_OFFLINE=1 uv run python scripts/evidence_provenance_proof.py
UV_OFFLINE=1 uv run python scripts/release_presentation_audit.py --root .
```

With explicit offline-capable Python 3.12 and 3.13 interpreters, also run:

- `scripts/consumer_source_pack_proof.py` with `--candidate-output`;
- `scripts/release_consumer_smoke.py` against the exact receipt-bound wheel;
- `scripts/compiled_library_export_proof.py`; and
- every canonical E1 through E3-E validator required by the identity closure.

CI must finish successfully for the exact reviewed PR head before merge. Post-merge checks on the
exact merge SHA must also finish successfully before publication.

## Failure Handling

- Version, documentation, wheel, receipt, or proof digest mismatch: stop; do not tag.
- Dirty source or changed HEAD during proof: invalidate evidence and rerun from a clean commit.
- Evaluation semantic drift: restore the transaction or recover checked-in artifacts, then stop.
- CI failure: diagnose and repair on the branch, review the new diff, and rebind checks to the new
  reviewed HEAD.
- Tag already exists with a different object or target: stop without moving it.
- Release already exists with conflicting state: stop without overwriting it.
- Archive smoke failure: keep the published facts intact, do not claim completion, and report the
  exact failing gate for a separately reviewed repair.

## Acceptance Criteria

The release is complete only when:

- package, module, metadata, wheel, docs, smoke, and tag all agree on `0.1.3`;
- the reviewed PR is merged with all required checks successful;
- the exact merged `main` commit passes the full repository and dual-interpreter evidence chain;
- the candidate receipt and compiled-export proof bind the same wheel SHA-256;
- canonical evaluation validators pass without semantic drift;
- OCR historical evidence remains byte-identical and accurately bounded;
- annotated tag `v0.1.3` targets the exact verified commit;
- the GitHub Release is public, non-draft, non-prerelease, and has no extra assets;
- the public source archive smoke passes;
- durable post-release documentation records actual, not anticipated, publication facts; and
- task-owned clean branches/worktrees are safely reconciled without touching unrelated evidence
  worktrees or operator-owned external caches.

## Publication Boundary

Publication authorization for this design covers the release-candidate PR, merge after all gates,
annotated tag, GitHub Release, public source archive smoke, docs-only post-release closeout, and
task-owned safe cleanup. It does not cover PyPI, another registry, deployment, model upload, or
production OCR implementation.
